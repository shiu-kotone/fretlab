# FretLab 仕上げ改修指示書(リリース品質化)

- 版: 1.0(2026-07-04 全機能実装完了後の全体レビューに基づく)
- 想定読者: 実装を担当するAIエージェント(Claude Sonnet)
- 目的: SPEC.md 全8フェーズ実装済みの現状を、**一般公開できるスマホアプリ品質**に引き上げる

---

## 0. 作業ルール(全フェーズ共通・厳守)

1. 作業前にリポジトリ直下の `SPEC.md` と本書を熟読すること
2. 仕様に曖昧さがあれば**実装前に解釈を明示**すること
3. 既存の392件のテストを壊さないこと。純粋ロジックを追加した場合はユニットテストを追加すること
4. 各フェーズ完了時に `npm test` / `npm run build` に加え、**Playwright(headless Chromium)で
   ダーク/ライト両テーマ・モバイル幅(390px)のスクリーンショット確認**を行うこと
   (preview は `npm run preview -- --port 4173`、base パスは `/fretlab/`)
5. 完了報告は「変更概要 / 確認手順 / 既知の制限」の形式で行い、
   **push は必ずユーザーの明示的な確認を待つこと**
6. 1フェーズ=1コミット(1PR)。R1 → R2 → R3 → R4 の順に依頼される想定

### 検証で使える既知のテクニック(過去フェーズで確立済み)
- 音の発音確認: `MessagePort.prototype.postMessage` をパッチして `type:'pluck'` を数える
- クリック音確認: `BaseAudioContext.prototype.createOscillator` をパッチして呼び出し数を数える
- 実タッチ検証: CDP `Input.dispatchTouchEvent` + `hasTouch: true` コンテキスト
- マイク系: `--use-fake-device-for-media-stream --use-fake-ui-for-media-stream`
- Chromiumバイナリは `$LOCALAPPDATA/ms-playwright` にキャッシュ済み(`playwright-core`のみ都度インストール)

## Phase R1: リリースブロッカー修正(最優先)

公開した瞬間にユーザーが気づく致命的問題。すべて確認済みの実バグ。

### R1-1. アプリアイコンが存在しない
**現状**: `public/` には audio-worklets しかない。`vite.config.ts` は `includeAssets: ['favicon.svg']`
を参照しているが**ファイルの実体がない**。manifest は `icons: []`。apple-touch-icon もない。
→ iOSでホーム画面に追加するとページのスクリーンショットがアイコンになる。リリース不可能。

**対応**:
1. `public/favicon.svg` を新規作成する。デザイン指針:
   - 角丸正方形、背景 `#16130F`(--bg)、モチーフはアンバー `#E8A33D` の指板
     (例: 縦3本の弦 + 横2本のフレット線 + ポジションドット1個。過剰装飾禁止、小サイズで判別できる太い線)
2. そのSVGから `public/icons/icon-192.png` / `icon-512.png` / `icon-512-maskable.png`(セーフゾーン
   内80%に収めた版)/ `apple-touch-icon.png`(180×180)を生成する。
   生成は一時的なNodeスクリプト+playwright-coreでSVGをラスタライズしてよい(スクリプトはコミットしない)
3. `vite.config.ts` の manifest に `icons`(192/512/maskable)と `description` と `lang: 'ja'` を追加
4. `index.html` に `<link rel="icon">` `<link rel="apple-touch-icon">` `<meta name="description">` を追加

**受け入れ**: build 後の `dist/manifest.webmanifest` に icons が入っている。
`dist/icons/*.png` が存在し、192/512 のサイズが正しい。

### R1-2. ライトテーマの配色破綻(コントラスト不足で文字がほぼ読めない)
**現状**: `src/index.css` のライトテーマ上書きが `--bg / --surface / --line` の3つだけ。
文字色 `--string: #c8cdd4`(明るい銀)が白サーフェス上にそのまま使われ、**ボタン文字が
ほぼ視認不能**(スクリーンショットで確認済み)。`--accent` も白地では薄い。度数カラー5色も未調整。

**対応**: `:root[data-theme='light']` で**全トークン**を定義し直す。開始値の提案:
```css
--bg: #f4efe8;  --surface: #ffffff;  --line: #d8cec0;
--string: #453d33;          /* 本文・ボタン文字: 白地で4.5:1以上 */
--accent: #b26a00;          /* アンバーを暗く: 白地で3:1以上(UI/大文字用) */
--ok: #2e7d36;  --warn: #c23327;
--degree-third: #b8503f;  --degree-fifth: #2f6ea8;
--degree-seventh: #6f4fb0;  --degree-tension: #5c6874;
```
注意: 多くのコンポーネントがアクセント塗りボタンの文字に `color: var(--bg)` を使っている。
ライトで --accent を暗くすれば「暗いアンバー地 + クリーム文字」となり成立する(要目視確認)。

**受け入れ**: WCAGコントラスト比を実際に計算して報告する
(--string/--surface ≥ 4.5、--accent/--surface ≥ 3.0、--bg文字/--accent地 ≥ 4.5)。
ライトテーマで全5タブのスクリーンショットを撮り、視認性を確認する。

### R1-3. 進行再生がタブ切替で停止する(SPEC §3.1 違反)
**現状**: SPEC §3.1「タブ切替時も再生中の音(メトロノーム・**進行再生**)は止めない」に対し、
`App.tsx` はコードタブを `{tab === 'chords' && <ChordTabView />}` でアンマウントし、
`useProgressionEngine` の `useEffect(() => stop)` が再生を止めてしまう。

**対応**: MetronomeView と同じ手法で、ChordTabView を `display:none` による常時マウントに変更する
(`App.tsx` の該当行をメトロノームと同じ形に)。あわせて:
- `TabBar` の `badges` に進行再生中の `chords: true` を渡す
  (進行エンジンの isPlaying をグローバルに知る必要がある。`playbackCoordinatorStore` の
  `activeOwner === 'progression'` で判定するのが最小改修)
- 副作用に注意: ChordLibraryView の `useActivityTimeTracker('chords')` が常時マウントで
  回りっぱなしになってはならない。**タブがアクティブな時のみ計測**されるよう、
  `useNavigationStore` の activeTab を見てトラッカーを有効化する引数を追加すること
  (指板・チューナーはタブ切替でアンマウントされるので従来通りで良い)

**受け入れ**: Playwrightで進行再生開始→リズムタブへ移動→pluckメッセージが継続して発火する
ことを確認。コードタブのバッジ点灯/消灯も確認。activityの二重計上がないこと。

### R1-4. チューナーにデバッグ表示が常時出ている
**現状**: `MicTunerPanel.tsx` 末尾に `lvl: 0.0123 / raw: 82.4Hz / conf: 0.912` という
開発用デバッグ行が常時表示されている。

**対応**: この表示を削除する(`debug` の仕組み自体は残してよいが、UIからは消す)。

### R1-5. 指板の開放弦ラベルが二重表示される
**現状**: `Fretboard.tsx` で、開放弦ゾーンのラベル(x=OPEN_ZONE_WIDTH/2, y=弦+8)と、
fret=0 のノートラベル(labelMode='natural' 時、同じx、y=弦−18)が**両方**描画され、
E/E、B/B…と重なって見える(スクリーンショットで確認済み)。

**対応**: fret=0 をベースラベル表示の対象から除外する(開放弦ゾーンのラベルに一本化)。

### R1-6. ヘッダーのタップ挙動が表示と食い違う
**現状**: ヘッダー全体が `aria-label="設定を開く"` でラボタブへ遷移するが、ラボタブの
初期セグメントは「録音」なので設定は開かない。またヘッダー全域がタップ可能で誤タップしやすい。

**対応**: LabView のセグメントストアに「設定セグメントを開いて遷移する」導線を追加し、
ヘッダーからはそれを呼ぶ。タップ範囲はチューニング名+左利きバッジ部分(右側)に限定し、
設定アイコン(歯車のインラインSVG)を添えて視覚的に示す。

## Phase R2: UIの洗練(見た目の統一とタッチ感)

### R2-1. 共通UIコンポーネントの導入(最重要)
**現状**: `smallButtonStyle` 等のインラインスタイル定義がファイルごとにコピペされ10箇所以上重複。
`:active` 状態・transition が一切なく、**タップしても視覚反応がない**。

**対応**: `src/components/ui/` を新設し、以下を実装して全画面を置き換える:
- `Button`(variant: primary / secondary / danger、`:active` で scale(0.97)+輝度変化、
  transition 120ms、disabled 状態、最小44×44pt保証)
- `Chip`(選択チップ。ルート/タイプ/拍子等の横スクロール列で使用中の形を統一)
- `SegmentedControl`(ライブラリ/進行、録音/記録/設定、マイク/音出し の3箇所で重複実装
  されている分割ボタンを統一)
- `Slider`(最低限 `accent-color: var(--accent)` を全 range input に適用。可能なら
  トラック/サムをCSSでテーマ色に統一)
- `Toggle`(ネイティブcheckboxをiOS風トグルスイッチに置換。44pt以上のタップ領域)
- `ConfirmDialog`(R2-4で使用)
置き換えは機械的に行い、**見た目の回帰をスクリーンショットで全タブ比較**すること。
CSSは index.css にクラスとして追加してよい(CSS Modules導入は不要)。

### R2-2. タブバーにアイコンを追加
**現状**: 文字のみ(SPEC §3.1 はアイコン想定)。
**対応**: 5タブにインラインSVGアイコン(フレットボード/ダイアグラム/メトロノーム/音叉/フラスコ等、
線画2px・currentColor)を追加し、アイコン+ラベルの縦積みにする。タブバー高さは safe-area 込みで調整。

### R2-3. 44ptタップターゲット監査
**現状の違反箇所(確認済み)**: MicTunerPanel の A4 ±ボタン(32px)、コードライブラリの
ボイシングドット(10px)、各所のネイティブcheckbox。
**対応**: 全タップ要素を44×44pt以上にする(視覚サイズは小さくても `padding` で当たり判定を拡大)。
ボイシングドットはドット自体を大きくするのではなく、タップ領域44pxのボタンの中に10pxのドットを描く。

### R2-4. ネイティブ confirm をアプリ内モーダルへ
**現状**: 削除確認・インポート確認・全消去二重確認がすべて `window.confirm`(OS素のダイアログ)。
**対応**: `ConfirmDialog` コンポーネント(オーバーレイ+カード、danger色の確定ボタン)に置換。
二重確認は「1つのダイアログ内で確認文言を2段階に変える」方式でよい。
対象: 録音削除 / 進行削除 / インポート置き換え確認 / 練習記録全消去 / 録音全消去。

### R2-5. フォント指定の解消
**現状**: `--font-display: 'Archivo', 'Bricolage Grotesque', ...` と指定しているが
**どちらのフォントも読み込んでいない**(常にシステムフォントにフォールバック)。
**対応**: どちらかを選ぶ(解釈宣言すること):
- (a) 数字グリフ中心のサブセットwoff2(数KB)を `src/assets/` に同梱し `@font-face` で読み込む
- (b) 読み込まないフォント名を削除し、システムフォント+`font-weight: 700`/`letter-spacing`
  の調整で大型数字(BPM・チューナー音名)の見た目を整える
サイズ予算(gzip 500KB)と オフライン要件を踏まえると (b) が安全。

### R2-6. チューナー初期画面の空白解消
**現状**: マイク許可前は画面上部にボタン1つだけで下が全て空(スクリーンショットで確認済み)。
**対応**: 許可前も CentsMeter と6弦ボタンを無効状態(opacity 0.4・操作不可)でプレビュー表示し、
中央に許可ボタンを重ねる。何ができる画面なのかを一目で伝える。

### R2-7. コードライブラリの初期表示改善
**現状**: ルート1行+タイプ4行のチップで画面上半分が埋まり、**ストラムボタンが初期表示で画面外**。
**対応**(いずれか、解釈宣言の上):
- (a) タイプ選択を「グループタブ(基本/セブンス/テンション/その他)+1行チップ」に畳んで縦を節約
- (b) ダイアグラム+再生ボタンを上、選択UIを下に並べ替え
ストラム/アルペジオは長押しの発見性が低いので、「ストラム」「アルペジオ」の2ボタンに分割してよい
(長押し併用は残しても良い)。

### R2-8. 記録画面のグラフ軸ラベル
**現状**: ヒートマップに月・曜日の目印がなく、どのマスがいつか分からない。
**対応**: ヒートマップ左に曜日(月・木のみ等間引き)、下に月境界ラベルを追加。
週間棒グラフはバー上に分数を表示(0分は省略)。

## Phase R3: 音の質感向上

### R3-1. ストラムの人間味
**現状**: 全弦が同一velocity(アクセント0.9/通常0.7)・同一brightness(0.5)・完全等間隔の
ステレオタイプな機械演奏。
**対応**: `useProgressionEngine.playStrumAction` と `useChordPlayback` に:
- 弦ごとの velocity 揺らぎ(±0.06程度のランダム)
- タイミング揺らぎ(±4ms程度)
- ダウンは低音弦をやや強く、アップは高音弦をやや強く(0.05程度の傾斜)
- 巻弦(6-4弦)は brightness を 0.45、プレーン弦は 0.55 に
乱数はテスト可能性のため注入可能な形(引数 or シード)にし、揺らぎ計算の純粋関数に
ユニットテストを付けること。

### R3-2. コード切替時の濁り解消
**現状**: 進行再生でコードが変わっても前コードの弦が鳴り残り(サステイン2秒)、
テンポによっては和音が濁る。
**対応**: `karplus-strong-processor.js` に `{type:'dampAll', releaseSeconds:0.05}` メッセージを
追加し、全voiceを短いリリースで消音できるようにする。`KarplusStrongSynth` に `dampAll()` を生やし、
進行エンジンで**新しいコードの最初のストラム直前**に呼ぶ(同一コード内の連続ストラムでは呼ばない。
現在の `lastSegmentKeyRef` 相当の「セグメントが変わった瞬間」で判定)。
アルペジオパターンでは呼ばない(鳴り重なりが本質のため)。

### R3-3. マスターリミッターの追加
**現状**: ギターバスは compressor 経由だが、クリックバスは destination 直結。
両者が同時に鳴ると合算でクリップし得る。
**対応**: `AudioEngine.ts` で両バスの合流先に master GainNode + DynamicsCompressor
(threshold -3dB, ratio 20, knee 0 のリミッター設定)を1段追加し、
destination への直結をやめて全音声がここを通るようにする。

**受け入れ(R3全体)**: 既存テスト+新規純粋関数テストがパスし、Playwrightで
カノン進行を2小節再生して pluck/click 発火数が改修前と同等であること
(dampAll によって発音が欠けないこと)を確認。

## Phase R4: 操作性・堅牢性の仕上げ

### R4-1. ボイシングのスワイプ切替(SPEC §5.2 の未実装分)
ChordDiagram 上の水平スワイプ(pointerdown→pointerup の水平移動が40px超・垂直移動がそれ未満)で
前後のボイシングに切替える。既存のドット/前後ボタンは残す。
左利きモードでもスワイプ方向の意味は変えない(表示が反転しても「左スワイプ=次」で統一し、解釈を宣言)。

### R4-2. バックグラウンド復帰時の安全な復元(SPEC §7)
`visibilitychange` で visible に戻った際:
- AudioContext が suspended なら resume する(共通処理を AudioEngine に追加し App.tsx で購読)
- Wake Lock は自動解放されるため、メトロノーム/進行が再生中なら再取得する
iOSは画面ロックでタイマーもAudioも止まるが、復帰時にUIと再生状態が矛盾しないこと
(スケジューラが動いているのに音が出ない状態を作らない)を目標とする。

### R4-3. 録音リストの操作改善
録音行に再生時間・サイズは表示済み。追加で:
- 行の展開中に他の行をタップしたら前の行を閉じる(現状は同時に1つだが確認)
- 削除を ConfirmDialog 化(R2-4と重複するのでR2で対応済みならスキップ)

### R4-4. SW更新の通知(任意・小)
`registerType: 'autoUpdate'` のままでよいが、更新適用後の初回起動時に
「アプリを更新しました(vX.Y.Z)」のトーストを1回出す(localStorage に前回バージョンを記録して比較)。

### R4-5. package.json バージョン更新
リリース準備として `version` を `1.0.0` に上げる(設定画面のバージョン表示に反映される)。

## 全フェーズ共通の受け入れ基準

1. `npm test` 全件パス(既存392件+追加分)、`npm run build` 成功、バンドル gzip 500KB 以下
2. ダーク/ライト両テーマ × 全5タブのスクリーンショットに視認性・レイアウト崩れがない
3. 左利きモード+カスタムチューニング設定状態で全タブを巡回してコンソールエラーゼロ
4. コミットメッセージは既存の慣例(英語・`Co-Authored-By: Claude Sonnet 5` 付き)に従う
